# 云层统一接口 v0

## 文档状态

- 状态：Draft v0
- 目标：冻结前端、Go、Runtime、硬件的最小联调接口
- 主线：设备控制闭环

## 接口分工

| 调用方向         | 协议        | 作用               |
| ------------ | --------- | ---------------- |
| 前端 → Go      | HTTP      | 发起动作、查询设备、触发 OTA |
| Go → 前端      | WebSocket | 推送状态、执行结果、错误     |
| Go → Runtime | gRPC      | 获取动作决策或参数转译结果    |
| Go ↔ 硬件      | MQTT      | 命令下发、状态事件回传      |
| 硬件 OTA 下载    | HTTP      | 下载固件或升级包         |

## HTTP 接口

### 1. 执行动作

```http
POST /api/v1/action/execute
Content-Type: application/json
```

请求体：

```json
{
  "version": "v0",
  "type": "action",
  "trace_id": "9f0e7b14-bf58-4b6a-9a2a-8f4e7f8e6001",
  "source": "frontend:web",
  "target": "runtime:default",
  "action": "shake_head",
  "params": {
    "device_id": "esp32-001",
    "speed": 100
  },
  "data": {},
  "timestamp": 1710000000
}
```

响应体：

```json
{
  "accepted": true,
  "trace_id": "9f0e7b14-bf58-4b6a-9a2a-8f4e7f8e6001",
  "device_id": "esp32-001",
  "runtime_action": "shake_head",
  "status": "queued"
}
```

说明：

- 前端通过 `params.device_id` 指明目标设备
- Go 校验参数后调用 Runtime
- Go 成功转发 MQTT 后返回 `accepted=true`
- 最终执行结果通过 WebSocket 推送，不要求 HTTP 长连接等待设备完成

### 2. 查询设备状态

```http
GET /api/v1/device/{device_id}
```

响应体：

```json
{
  "device_id": "esp32-001",
  "online": true,
  "last_seen_at": 1710000003,
  "last_state": {
    "status": "done",
    "trace_id": "9f0e7b14-bf58-4b6a-9a2a-8f4e7f8e6001"
  }
}
```

### 3. 触发 OTA

```http
POST /api/v1/device/ota
Content-Type: application/json
```

请求体：

```json
{
  "device_id": "esp32-001",
  "firmware_version": "v0.1.0",
  "artifact_url": "https://example.com/firmware/v0.1.0.bin"
}
```

响应体：

```json
{
  "accepted": true,
  "device_id": "esp32-001",
  "status": "scheduled"
}
```

## gRPC 接口

```proto
syntax = "proto3";

package rak;

service RuntimeService {
  rpc Execute (ActionRequest) returns (ActionResponse);
}

message ActionRequest {
  string version = 1;
  string trace_id = 2;
  string source = 3;
  string target = 4;
  string state = 5;
  repeated string available_actions = 6;
  string action = 7;
  string params_json = 8;
}

message ActionResponse {
  string version = 1;
  string trace_id = 2;
  string action = 3;
  string params_json = 4;
  string status = 5;
  string error_code = 6;
  string error_message = 7;
}
```

### gRPC 语义

- Go 是调用方，Runtime 是被调方
- `state` 承载当前上下文摘要，例如 `user_click_button`
- `available_actions` 允许 Go 把当前设备可执行动作显式传给 Runtime
- `params_json` 用于承载动态参数，内容必须能还原为 RakMessage `params`
- `status=ok` 表示返回可执行动作
- `status=error` 时，Go 不得继续下发 MQTT 命令

## 错误码

| code                      | 含义           | 处理建议                     |
| ------------------------- | ------------ | ------------------------ |
| INVALID\_PARAMS           | 参数不合法        | Go 直接返回前端                |
| DEVICE\_OFFLINE           | 设备离线         | Go 不调用 Runtime 或不下发 MQTT |
| RUNTIME\_UNAVAILABLE      | Runtime 不可用  | Go 返回错误并记录 trace         |
| RUNTIME\_DECISION\_FAILED | Runtime 决策失败 | Go 不下发设备命令               |
| MQTT\_PUBLISH\_FAILED     | 命令下发失败       | Go 返回失败并推送错误事件           |
| DEVICE\_TIMEOUT           | 设备超时未回传      | Go 推送超时错误给前端             |

## 超时与幂等

- HTTP 请求收到后，Go 应在短时内返回受理结果，不阻塞等待设备完成
- Go 调 Runtime 超时后，当前 `trace_id` 进入失败态
- 同一 `trace_id` 的重复执行请求默认按幂等处理，不重复下发第二次设备命令
- 设备状态回传必须带回原 `trace_id`，否则 Go 只作为普通设备状态记录，不回填具体动作链路

## 联调顺序

1. 前端与 Go 先对齐 HTTP 请求体和 WebSocket 推送体
2. Go 与 Runtime 冻结 gRPC 请求响应
3. Go 与硬件冻结 MQTT topic 与 payload
4. 四方用同一个 `trace_id` 做端到端联调验证
