# 技术环境标准

## 文档状态

- 状态：Draft v0
- 目标：为两周 MVP 提供最小可执行环境基线
- 原则：只装当前角色必需依赖，不一次性安装全部未来能力

## 通用原则

- Node 项目统一使用 `pnpm`
- Python 项目统一使用 `uv` 或 `conda`，不使用系统 Python 环境
- 正式标准不保留 notebook 安装输出、错误日志和实验命令残留
- 先满足联调，再考虑训练、推理、向量库等扩展能力

## 角色环境

### 1. 前端

| 项目 | 标准 |
| --- | --- |
| 包管理 | pnpm |
| 目标职责 | 最小操作界面、动作发起、状态展示 |
| 必需能力 | HTTP 请求、WebSocket 接收、基础页面开发 |

最小要求：

- 能发起 `POST /api/v1/action/execute`
- 能订阅 `/ws`
- 能展示设备在线状态、动作执行状态、错误消息

### 2. Go 云层

| 项目 | 标准 |
| --- | --- |
| 语言 | Go |
| 目标职责 | API 入口、Runtime 调用、MQTT 转发、状态推送 |
| 必需能力 | HTTP 服务、WebSocket、gRPC Client、MQTT Client |

最小要求：

- 能接收前端动作请求
- 能调用 Runtime gRPC 接口
- 能向 MQTT 发布命令并订阅设备状态
- 能把状态回推给前端

### 3. AI Runtime

| 项目 | 标准 |
| --- | --- |
| 语言 | Python |
| 环境方式 | 本轮优先 conda |
| 原因 | 现有素材已基于 conda，且后续若接 PyTorch CUDA 更平滑 |
| 目标职责 | Fake Runtime 最小决策服务 |

本轮建议：

- 先准备独立 Python 环境
- 只安装支撑 Fake Runtime 的最小依赖
- 将大模型、训练、向量库相关依赖降级为扩展项

扩展依赖说明：

- 若后续接入真实 AI Runtime，再按需要补充 PyTorch、transformers、sentence-transformers 等
- 当前 notebook 中一次性安装大量包的做法不作为正式基线

### 4. 硬件联调

| 项目 | 标准 |
| --- | --- |
| 目标职责 | 设备订阅命令、执行动作、回传状态 |
| 必需能力 | MQTT 连接、JSON 解析、基础 OTA 接入 |
| 联调方式 | 先跑一台目标设备即可，不要求同时覆盖多型号 |

最小要求：

- 能订阅 `rak/{device_id}/cmd`
- 能向 `rak/{device_id}/state` 回传标准 JSON
- 能上报心跳与在线状态

## 环境分层

### 必需依赖

- 前端页面与实时展示相关依赖
- Go 的 HTTP、gRPC、MQTT 能力
- Runtime 的最小 Python 服务能力
- 硬件侧 MQTT 与 JSON 处理能力

### 扩展依赖

- PyTorch CUDA
- 向量数据库
- LangChain LangGraph
- Kafka
- 训练与强化学习相关工具
- 记忆系统相关中间件

## 对现有素材的清洗原则

- `example/技术环境标准.ipynb` 中的实验命令可作为参考来源，但不直接视为标准
- 安装报错和混合依赖列表不进入正式文档
- 把环境说明改为角色视角和职责视角，而不是工具堆叠视角

## 最小验收

- 前端能连上 Go 并收到 WebSocket 状态
- Go 能成功调用 Runtime 并发布 MQTT
- Runtime 能稳定返回动作结果
- 硬件能执行至少一个动作并回传完成状态
